Portals 3.0 · Documentation

How Portals actually works

A complete walkthrough of the concept demo: the idea behind it, exactly how the content model and renderer work, what every surface does, and a suggested order for presenting it. Written to cover every possibility in the demo — read once for full context, then kept as a reference.

Prepared for · Internal review Subject · AI-driven corporate CMS concept Status · Clickable prototype, representative data

01 — The idea

Two bets, not one.

Portals rests on two separate claims. The first is about how pages get made. The second is about who's allowed to do what. Everything documented after this section is one of these two bets, worked out in detail.

1

Content and design are the same object, rendered two ways

A page is not a file that gets edited — it's the output of a function. render(content, theme). Change the content, the page updates. Change the theme, the exact same content re-lays-out into a completely different design. Nothing is ever hand-drawn twice, and no one ever touches a layout directly — because a layout isn't a property any node has. Section 03 documents exactly how this function is built.

2

AI proposes, a human approves — everywhere, every time

AI turns raw text into structure and drafts refreshes to existing content. It never publishes anything on its own. Every AI-authored change carries a visible provenance badge until a person reviews it, and every refresh is staged as an approve/reject diff before it commits. Sections 06 and 07 show this gate in the ingest flow and the maintenance workflow respectively; section 09 collects every rule that governs it in one table.

A detail worth noticing

The split even runs through the typography. Rendered pages are set in adidasFG — the same condensed brand face a real adidas page would use — while every tool surface (Workspace, Maintenance, Preview's own chrome) is set in the system UI font. It's a small thing, but it means you can always tell, at a glance, whether you're looking at output or at the tool.

02 — The data model

Everything is a node, one graph.

Four node types — Site, Page, Section, Block — nest in a strict tree (one parent each). Independently, any node can carry refs: reference edges to any other node, regardless of where either one sits in that tree. The tree is structure; refs are relationships. That distinction is load-bearing everywhere else in the product.

Structural vs. reference edges

Every node has exactly one structural edge (its parentId) — that's the tree. Independently, refs[] draws reference edges to any other node in the graph, in either direction, proposed by AI during ingest or drawn by hand. When a reference points at a Block, it isn't just "related" — its content is embedded: pulled onto the host page live, read fresh every time that page renders (the 2030 Targets example above). The visual and interaction treatment of both edge types lives in §05.

03 — The renderer & theme grammar

One renderer. Three grammars.

There is exactly one render path in the whole product: composeTheme(theme, content) takes a fixed, ordered sequence of block kinds for the active theme, and keeps only the ones the node actually has content for. Nothing about a theme is a color palette — it's an order of operations.

Two supporting mechanisms, introduced once

WireframeSketch — a cheap grayscale skeleton driven by the exact same composeTheme() sequence as the real renderer. It's not a fake finished design; it's the honest, structure-only version of the same grammar, used wherever a node hasn't been rendered yet (§06 workspace hovercards, §06 ingest's live structure preview).

BlockSlot — every rendered block is wrapped in this hook. It can show a highlight outline (used for Maintenance's visual diff, §07) and, in Preview, becomes clickable to open that block's editor (§08).

04 — Chrome & the three surfaces

The shell around all three rooms.

Two persistent bars wrap every surface: a top nav that switches rooms, and — in Workspace — a toolbar for acting on the graph directly. Both are worth understanding before touring each surface, since they appear, unglossed, in every screenshot from here on.

The Workspace toolbar

Breadcrumb reads "SITE / {selected node}". Three icon tools sit centered — Add node (always available), Delete node (only when a non-root node is selected), Preview page (only when something is selected). On the right: Save (flashes "Saved ✓" for 1.4s, cosmetic) and Publish.

Add / Delete node

Add creates a new Section named "New section," parented under the current selection, inheriting its context and theme, marked draft (not yet rendered) — and selects it immediately. Delete removes the selected node (the root can't be deleted), lifts any orphaned children up to its own parent so subtrees are never lost, and strips the deleted id out of every other node's references.

The publish flow

02

03
Published

The Publish button disables and reads "Published"; the badge flips amber Staging → green Live, instantly.

04
Any subsequent edit reverts it

Rename a node, flip a theme, edit a paragraph, approve a refresh, sync a stat — literally any mutation anywhere in the product sets the badge straight back to Staging and re-enables Publish. This is enforced automatically on every mutating action, not a manual step someone has to remember.

05 — The workspace

The canvas is the site, not a diagram of it.

Every card is a real node; every line is a real relationship. The canvas supports three groups of behavior: getting around it, reading its structure, and directly manipulating the graph.

Getting around

Zoom — mouse wheel, toward the cursor, plus an on-canvas −/%/+/Fit cluster (0.3×–2×; Fit computes a bounding box of visible nodes and centers to it). Pan — click-drag the empty canvas background, which carries a dotted grid. Collapse / expand — any node with children shows a caret pill ("–" to collapse, "+3" to expand with a child count); collapsing hides the whole descendant subtree from view, and the layout self-heals since positions are always auto-computed, never authored.

Reading structure and relationships

Structural edges (parent → child) draw as solid dark elbowed "bus" lines merging siblings into a shared trunk. Reference edges draw dashed violet, routed through parallel lanes in the left gutter so multiple references never overlap. Hovering any node dims everything except its direct neighborhood (parent, children, references, referencing nodes) and highlights just the touching edges. Clicking an edge selects it and reveals a small red "×" to delete it. A small "ⓘ" legend, top-right, expands on hover into a full edges/provenance color key.

Direct manipulation

Each card exposes a small circular drag-to-link handle on its right edge; dragging from it draws a live rubber-band line, and releasing over another card creates a new reference edge on the spot — link curation by hand, no dialog. Hovering a node for 220ms pops a rich hovercard: a live scaled-down render of the actual page (or a WireframeSketch if it's still a draft), its provenance badge, lead text, live bindings, and quick Blocks/Links/Live counters.

06 — Ingest

Raw text in, structure out.

One floating input, two very different things it can do with the same text: propose a whole new branch of the site, or read a natural-language command about the graph that already exists.

(Only seen if the modal is opened empty.) A textarea and one explanation: the AI proposes where this fits in the existing site as a structure, not a finished page. Button: "Generate structure."

04

Step 2 · Adjust structure

Left: an "Attaches under" dropdown (defaults to Newsroom) plus an editable tree of proposed nodes — rename, change context/theme/parent, or delete (orphans lift to the deleted node's parent). Right: a sticky live WireframeSketch preview of whichever node is hovered or focused, captioned "Structure only — hover a node or change its theme to update."

06
Step 3 · Connect

AI-suggested cross-links into the existing graph, each a toggle row with a plain-language reason (e.g. "Shares the carbon-neutral materials programme") — every suggestion starts checked; uncheck any you don't want.

Nodes land on the canvas immediately as drafts (AI-generated provenance). A "Generating pages…" overlay runs ~1.3s, then every node flips to rendered and Preview opens automatically on the root of what you just placed. There is no manual per-node "Generate" click in this path.

07 — Maintenance

The same graph, tinted by health.

Four signals, computed live from real fields on the node — never a separately stored rating. Every one of them is fixable in a single click.

The four health signals

SignalDetected whenActionExample reason shown
Data driftedA bound stat's synced value ≠ its live source value.Sync"Net sales (Q3): €6.6B → €6.9B · Finance DB"
Needs reviewOrigin is AI-generated.Review"AI-generated — not yet reviewed by a human"
Stale contentMore than 21 days since last update.Refresh"Last updated {updatedAt}"
Draft pageNot rendered yet (or mid-ingest).Generate"Page not generated yet"

Severity, most to least: drift > review > stale > draft — a node's "top issue" is whichever of these it has that ranks highest, and that's what colors its card everywhere (§05).

The KPI strip and attention list

A single Site Health percentage — flagged nodes ÷ total, inverted — sits beside filter chips for each signal (click to isolate, click again to clear). Below it, a prioritized "Needs attention" list: colored dot, node name, its parent, the concrete reason string, a Compare icon (drift and stale rows only), and the action button itself.

One-click resolution

Sync rewrites every drifted stat to its live value and marks the node Synced. Review is a pure sign-off — origin flips AI-generated → Human-edited with no content change. Refresh doesn't commit directly; it stages a Refresh diff (§08) for approval first. Generate renders a draft immediately and opens it in Preview. The Compare icon on drift/stale rows opens the same diff modal straight into its Visual mode.

The health-lensed graph

The same Canvas component, reused read-only — every editing affordance (drag-to-link, edge delete) is switched off, node borders/glows tint by top issue, and healthy nodes dim so problems visually pop out of the graph at a glance.

10 — Suggested demo script

A ten-minute run-through.

In the order you'd naturally hit them live. Every step below is a real click against the seeded demo data described in §02 — nothing here is staged specifically for presenting.

0:00 – 0:30
Orient.

Land on Workspace. Point out the Staging badge in the top nav and the three surface tabs.

0:30 – 1:30
The data model, live.

Open "Carbon Neutrality by 2030." Walk the Inspector top to bottom — context, theme, the populated content editor, references to Materials & Circularity and Supply Chain Standards.

1:30 – 2:30
The money shot.

Toolbar → Preview page. It renders under Editorial (hero → text → quote → images → stats → table). Flip the theme switcher to Technical on the same node — stats and the table jump to the top. This is render(content, theme), not a claim about it.

2:30 – 3:30
Compare mode.

Toggle "Compare themes" — Editorial and Technical, side by side, same content.

3:30 – 4:00
Newsletter channel.

Switch the channel toggle — same node, a fake mail client, a completely different medium.

4:00 – 5:00
Live data and drift.

Open "Third Quarter 2025 Results," show its Net sales binding, then jump to Maintenance — the attention list already flags it with the exact figure delta. Click Sync.

5:00 – 6:00
The approval gate.

Pick an AI-generated node from the same list, click Review, then open a Refresh diff in Fields mode and toggle to Visual mode. This is bet two, shown, not asserted.

6:00 – 7:30
Ingest, full flow.

Paste the FIFA World Cup press release into the floating bar. Walk Step 1 → Step 2 (adjust the tree, watch the live wireframe) → Step 3 (toggle one suggested cross-link off) → Place. Watch the generating overlay, then the new pages land as drafts.

9:15 – 10:00

11 — What's next

Prove it on one real page, then widen the surface.

Not a rebuild. The architecture in §01–§09 already proves the thesis. The next step is scoped to one page in this exact demo — Sustainability's carbon page — made real in three specific ways.

01 · The data

A real source

Its emissions stat gets bound to an actual internal number instead of a mock one.

02 · The AI

A real model

Its next refresh is drafted by a real AI call — reviewed and approved exactly as it is today.

03 · The proof

One page, end to end

If a curator can maintain that single page for real, the same graph is ready to carry the rest of the site.

What it takes

Not a headcount plan — three concrete things: a place to run (a real web page and database, not a local prototype), model access (an AI API key, to replace the canned responses with real ones), and dev support (someone to connect the two into what's already built).

Beyond the pilot

Once the one-page pilot is running, the same architecture extends outward in three directions that don't touch the core bet: deepen the connection layer — let a curator add and remove cross-links directly from the Inspector, with AI suggesting new ones as the graph grows; widen corporate data binding — swap mock sources for a real registry of live ones, with freshness tracking and one-click re-sync built in from the start; broaden oversight — batch review across many nodes at once, with richer before/after diffs, not just one section at a time. Content and design stay separate through all three — it's the same graph, doing more.
