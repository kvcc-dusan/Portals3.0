Portals 3.0 · Concept Walkthrough

How Portals
actually works

A working concept demo of an AI-driven CMS — the content graph is the source of truth; the rendered page is disposable output.

Status · Concept demo — no live integrations

01 — The idea

Stop authoring pages.
Curate a graph instead.

In a normal CMS, a page is the thing you make — you open an editor, arrange blocks, pick colors, and save a layout. Portals removes that step entirely. You maintain a content graph: structure, text, data, and the connections between them. The system computes every page from it, automatically, every time.

The bet

Separate what the content says from how it looks — completely — so a person never has to touch the second one.

02 — How it runs

A page isn't a file.
It's a function's output.

Nothing is hand-drawn twice. A page is recomputed on demand. Change the content, the page updates. Change the theme, the exact same content re-lays-out into a completely different design.

03 — The node model

Everything is a node.

A node is a small record: a name, a department it belongs to, a visual theme, some content fields, and a place in the hierarchy. Four node types exist, nesting exactly like a site's real structure does.

04 — What makes it trustworthy

Three things ride along with every node.

AttributeWhat it tracksWhy it matters
Corporate context Which department — Company, Sustainability, Investor Relations, Newsroom, Brands. Drives navigation and gives the AI a category to reason about.
Theme Which visual grammar renders it — Editorial, Technical, or Index. The design layer. Swappable per node, at any time, without touching content.
Provenance Who last touched it — AI-generated, human-edited, or synced from a source. The trust signal. Nothing AI-only and unreviewed is treated as finished.

05 — The workspace

Not a diagram of the site.
The site.

Every card on the canvas is a real node; every line is a real relationship. There are exactly two kinds of line, and they mean very different things.

Solid grey line

Structural edge

Parent → child. A page's children are its sections, a section's children are its blocks. Drag a card to move it in the hierarchy.

Dashed violet line

Reference edge

"These two are related," regardless of where either sits in the tree. Proposed by AI during ingest, or drawn by hand.

When it points at a block

It's not just related — it's embedded. The block's actual content is pulled onto the page it's referenced from, live, read fresh every time that page renders.

06 — Navigation

Three lenses on the exact same data.

Workspace

Author

The mind map — structure, content, connections, and the ingest bar that turns raw text into new nodes.

Maintenance

Keep correct

The same graph, tinted by health. Surfaces what's drifted, unreviewed, stale, or unfinished — fixable in one click.

Preview

See the output

Renders any node into an actual page, or a newsletter, so you can check exactly what's about to go live.

They are not three tools — they're three views on one graph. An edit made in one is instantly visible in the others.

07 — Human-in-the-loop

Enforced structurally,
not by convention.

There is no visual editor anywhere in the product. It cannot exist — because layout isn't a property a node has.

A person curates

  • Hierarchy — where a node sits, what it's parented to
  • Content — the actual words, figures, images
  • Connections — which nodes reference or embed which
  • Approval — every AI draft, every sync, waits for a click

The system generates

  • Every rendered page — in every theme, every time
  • Layout, spacing, composition — fully computed
  • Nothing hand-placed, nothing saved as a one-off

08 — The next step

Prove it on one real page.

Not a rebuild. Take one page that already exists in this demo — Sustainability's carbon page — and make three things about it real. Everything else stays exactly as it is today.

A real model

03 · The proof

One page, end to end

If a curator can maintain that single page for real, the same graph is ready to carry the rest of the site.
